# Finance Tracker — 产品开发需求文档 (PRD)

> 作者: 项目所有者
> 创建日期: 2026-05-03
> 最后修订: 2026-05-05
> 版本: v1.1

## 1. 背景与动机

我想要同时实现**记账**和**个人资金管理**两类需求。市面上没有一款现成软件能够同时满足，因此决定自研。

## 2. 核心功能需求

### 2.1 自动记账

#### (a) 手动导入 PDF
- 我可以在每月月底下载各种**银行卡**和**信用卡**的 PDF 账单并上传到软件中
- 软件需要**自动识别**账单内容
- 自动**对账目进行分类**

#### (b) 直接连接银行
- 通过银行提供的接口方案（例如 Open Banking / GoCardless / Nordigen 等）
- 让软件**实时连接银行账户**，获取最新的账单和记录
- 实现**完全自动化的记账**

#### (c) 分类功能
- 内容根据日常支出需求分类，**支持用户自行编辑定义**，**支持多级分类**
- 自动根据账单内容放到相应的列表
- **三层匹配管道（用户细化 2026-05-04）**：
  1. **关键词 / 规则匹配** —— 命中即归类（最快，零成本）
  2. **LLM 兜底** —— 关键词 miss 时调大模型；输入包含「分类知识库」（已有 rules + 关键词 + **历史用户备注**）作为推理线索
  3. **用户手动确认** —— LLM 仍不确定 → 进 inbox。用户确认时**可写一段备注**，备注会被持久化到知识库供下次 LLM 推理
- 如果不确定账单内容，把内容放进**"不确定"列表**等待用户确认
- 支持分类**可编辑**，以及用户**修改分类错误**的支出类别
- **带记忆功能**：用户每次确认 / 修改 → 反向写入知识库（关键词规则 + 备注），下次出现类似账单自动命中或被 LLM 用作上下文

#### (d) 分类知识库（用户细化 2026-05-04）
- 持续累积的"分类线索池"，由三类数据组成：
  - 规则关键词（已有 `categorization_rules` 表）
  - 用户在 inbox 确认时的**手写备注**（新需求，需扩展 schema）
  - 历史 (description → category) 映射（可从 `transactions` 表自动派生）
- **作用**：在调 LLM 时把相关条目拼到 prompt 里作为 few-shot 示例，提升 LLM 准确率
- 知识库本身可在 settings 页查看 / 编辑 / 导出

### 2.2 资产实时跟踪

我的资产种类较多且分布广泛，包括：
- 人民币 (CNY)
- A 股股票
- 欧洲股票
- 各类**加密货币**及加密钱包资产
  - **链上自动同步**（用户诉求 2026-05-03）：新建"加密钱包"账户时，**只需提供钱包公钥/地址**，软件应自动读取该地址在链上的资产组合（原生币 + ERC-20 / SPL token / UTXO 等），并写入持仓。无需手动逐项添加资产
- 欧元现金
- 黄金

由于这些标的价格波动较大，我很难实时掌握总资产的变化。因此需要软件能够：
- **连接市场接口**获取**汇率**和**资产价格**的实时波动数据
- **实时显示**当前所有资产的**总额**

### 2.3 个人资金 & 现金流记录与分析

需要一个个人储蓄/资金记录表，能够生成随时间变化的图表，涵盖：
- (a) 每月**收入**
- (b) 每月**支出**
- (c) 每月**储蓄**
- (d) 其他**现金流变化**

通过这些直观的图表，可以清晰地看到个人资金的流向和现金流的变化。

## 3. 产品形态

### 3.1 形式
- (a) **前期主要以 Web 页面**的形式呈现，提供一个直观的外部 UI 来展示各项数据
- (b) 后续可以考虑**开发 APP**，但现阶段以 **Web 端优先**

### 3.2 UI 功能模块
- (a) **PDF 上传区域**：支持用户直接上传文件
- (b) **个人资金变化图表**：直观展示总资产的波动情况
- (c) **记账功能模块**：展示每月支出明细及具体分类
  - **层级化交互（用户细化 2026-05-04）**：当前实现把所有交易平铺，**不直观**。需要重构为：
    - 用户先选择某一**一级类目**（住家 / 日常生活 / …）
    - 进入该类目后看其**二级类目列表 + 每月总额**（如 "房租 ¥3000 / 房屋维修 ¥0 / 清洁费 ¥120"）
    - 点二级类目展开看**该月该类目下的明细交易**
    - 顶部显示**该一级类目本月总额** + **占总支出比例**
  - 关键：让"消费结构"在视觉上立刻可见，而非"流水账 + 分类标签混杂"

## 4. Agent 接口支持

除了面向人类用户的 UI，该软件还必须提供针对 **Agent 的接口**，以便本地 Agent 能够：
- (a) **实时查看和调用**个人资金情况
- (b) 实现**自动化记账**：例如把 PDF 直接发给 Agent，Agent 能调用接口完成记账
- (c) **实时交互**：当我询问最近的资金情况时，Agent 能直观地反馈资金变化
- (d) 也就是说，这个软件在拥有用户 UI 的同时，**还要具备面向本地 Agent 的接口**（MCP / skills / API 等）

## 5. 数据存储

数据需要存储在**两个地方**：

### 5.1 本地数据库
- 可考虑 **SQLite 或 PostgreSQL**，具体选型由实现方决定

### 5.2 Notion 平台
- 同时把账目数据**存储在 Notion 上**
- Notion 上的具体存储形态可以在**稍后阶段再决定**

## 6. 非目标 / 暂不考虑

- 多用户、登录注册（个人单用户工具）
- 移动端 App（P2 之后再说）
- 投资建议、财务规划顾问（只做记录与展示，不做决策）

## 7. 后续补充需求（用户陆续提出）

> 这些是 v1.0 之后用户在使用过程中提出的细化需求，已编号入 ROADMAP。

### 7.1 跨账户转账识别（2026-05-05）
**问题**：N26 转 500 EUR 到 Amex 还款 + N26 转 1000 EUR 到 Revolut 储蓄 → 两边账单都上传 → 当前会双计 1500 EUR 月度支出（虚增）。

**要求**：
- 系统识别"内部转账"（不计入支出/收入）
- 同银行子账户（N26 Spaces / Revolut Pockets / Investing / Dream List 等）→ 不影响余额
- 跨银行转账（N26 → Revolut）→ 双方各自变化，整体 net 0

**实施约定**：
- 匹配窗口 ±3 天；自动配对置信度阈值 75
- 不为每个子账户建独立账户，但用户可在 settings 维护"子账户名清单" per account
- 同账户 / 同金额 / ±3d 自动配对（带描述相似度门槛）

### 7.2 IBAN 内部转账识别（2026-05-05）
**问题**：自己转账时账单写「账户持有人姓名」（自己名）而非银行名 → matcher 难识别。

**要求**：
- 账户字段加 IBAN
- 当转账描述里出现某账户的 IBAN → 高置信度判定为内部转账
- 内部转账不计入收入/支出

### 7.3 余额校准（2026-05-05）
**问题**：上传账单可能漏记或解析不完整 → 系统余额与银行真实余额不一致。

**要求**：UI 提供"校准余额"操作（设到目标值，系统自动建一笔 adjustment）。

### 7.4 高置信度自动通过 inbox + 同描述级联学习（2026-05-05）
**要求**：
- 命中规则的 tx 直接入账，不再要求"再确认一次"
- 用户改 1 笔 → 同 description 的所有兄弟 tx 自动归到同一分类（含已分类错的也跟着改）
- 跨 kind（如 expense → income）改时只改 seed，不级联（保护其他兄弟）

### 7.5 内联分类编辑 + 跨 kind 切换（2026-05-05）
**要求**：分类视图明细行 / transactions 列表行都能点 category 标弹下拉直接改；下拉支持跨 kind（支出/收入/转账）切换。

### 7.6 子账户清单管理（2026-05-05）
**要求**：在 settings 账户卡内提供"子账户名"编辑（用户可加 `Investing / Dream List / Saving`）→ PDF 解析时识别这些字符串自动归到 subaccount。
