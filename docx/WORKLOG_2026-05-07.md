# 2026-05-07 工作日志（功能盘点 + 架构审查输入）

## 一、今天交付的新功能（按依赖序）

### 1. CategoryScopeDialog（同名分类传播范围三选一）
**问题**：之前用户在 inbox 改分类，后端会自动级联到所有同描述兄弟行 + 学一条规则。但 PayPal 这种「同名不同用途」会被错误归类。

**新增**：
- 后端 `categorizer/engine.py`:
  - `count_similar_pending(db, seed_tx, category_id) → int`：预览级联会影响多少行
  - `derive_keyword_for_tx(tx) → (field, keyword)`：暴露 keyword 提取逻辑
  - `disable_rules_for_keyword(db, seed_tx) → int`：把同 keyword 的现有规则禁用
- 后端 `apply_to_similar_pending` / `count_similar_pending`：移除 `Transaction.type != "transfer"` 过滤（之前转账不能级联是 bug）
- 后端端点 `GET /transactions/{id}/similar-count?category_id=X`：弹窗预览数据源
- 后端 `update_transaction` / `confirm_inbox_item`：新增 query 参数 `apply_scope ∈ {all, single, never}`
  - `all`：学规则 + 级联（旧行为）
  - `single`：跳过两者
  - `never`：跳过 + 禁用现有同名规则
- 前端 `category-scope-dialog.tsx`（新文件，含 createPortal 解决 `<tr>` 嵌套问题）
- 前端 `inbox-panel.tsx`、`inline-category-picker.tsx`：分类有变化时弹出 dialog；备注输入也搬进 dialog

### 2. MarkTransferDialog 强制选转账分类 + 历史 transfer 重新入收件箱
**问题**：标记为转账时只设方向，不要求分类，导致很多转账无分类堆积。

**新增**：
- 后端 `MarkTransferIn` 加 `category_id`（必须 kind=transfer）
- 后端 `mark_as_transfer`：双边写入同一分类
- 后端 lifespan：`type='transfer' AND category_id IS NULL` 的活跃行 `is_pending=1` 重新入收件箱（首次 42 笔）
- 前端 dialog：强制下拉选转账分类（按一级分组 optgroup）

### 3. 子账户单边自动分类 + 修旧数据
**问题**：parser 识别子账户后只打 `metadata.subaccount=true`，不设 `category_id`。my migration 把这种行重新入收件箱→用户被骚扰。

**新增**：
- 后端 `ingestion/_backfill_transfer_category(tx, resolver)`：subaccount→内部储蓄；其他→跨行划转
- 后端 `accounts._reclassify_pending_for_subaccounts`：同步设 `category_id=内部储蓄`
- 后端 lifespan：扫 metadata.subaccount=true AND category=NULL 的，补内部储蓄（首次 19 笔）

### 4. 「信用卡还款」分类下线 → 合并到「跨行划转」
**理由**：TF Bank PDF 不显示对端还款，分类纠偏成本高，简化以减少误判。

**新增**：
- `seed.py` 移除该子分类，关键词全合并到跨行划转
- lifespan 3 步：rules 去重 + tx 改指 + category 删除（首次清 9 条重复规则）

### 5. 交易记录页面月份导航
**新增**：
- `lib/time-range.ts` 加 `monthDateRange / shiftPeriod / currentPeriod / recentPeriods`
- `transactions/page.tsx` 「list」tab 顶部 ◀ 月份选择 ▶ + 「全部时间」回退（自定义日期输入只在 全部时间 模式显示）

### 6. UnpairedTransfers 面板 + 手动绑定
**问题**：跨行 transfer 缺一边时（信用卡账单不显示还款）面板看不到。

**新增**：
- 后端 `GET /transactions/transfers/unpaired`：全局扫 `type=transfer AND counter_account_id IS NULL AND NOT subaccount AND NOT paired_with_tx_id`
- 后端 `MarkTransferIn` 加 `counter_account_id`
- 后端 `mark_as_transfer` 第 2 分支：counter_account 给定但无 counter_tx → 自动在对手账户创建镜像（synthetic_counterleg=true, external_id=`counterleg_of_X` 幂等）
- 前端 `TransferSuggestionsPanel` 拆两段：候选配对（旧）+ 未配对转账（新）

### 7. Counter-leg 智能绑定（避免重复）
**问题**：两边真腿都存在时，bind 任一边都造镜像 → 4 行（2 真 + 2 镜像）。

**新增**：
- 后端 `find_existing_counter_leg(src_tx, counter_account_id, window_days)`：在对手账户里搜同金额/币/±5 天/未配对/非 synthetic 的真腿
- `mark_as_transfer` counter_account 分支：先调它，命中→`pair_transactions`（双真腿）；失败→镜像
- 加 guard：源行已有 counter_account_id 则拒绝 422（避免 race 双绑）

### 8. matcher 双向兜底
**问题**：`find_transfer_pairs` 只看描述 from/to 关键词分方向，AMEX 风格「Payment received」没字样就漏。

**新增**：
- 优先级：metadata.transfer_direction → 描述关键词 → fallback 同时塞 outflows + inflows

### 9. 全局重新匹配按钮（page header）
**位置**：`transactions/page.tsx` header（手动记账按钮旁）

**新增**：
- 后端 `POST /system/refresh-matching` 9 步流水线：
  - Step -1：orphan-pointer 清理（counter_account 指向已删除/不存在对手 → 清掉指针）
  - Step 0：type 重判（`_classify_transfer` 跑 raw_description 拿子账户/跨行 hint）
  - Step 1：未分类 expense/income 跑规则
  - Step 2：同账户 ± 子账户对识别
  - Step 3：单边 IBAN 识别
  - Step 4：跨账户 ≥75 自动配对
  - Step 5：orphan 单边修复
  - Step 6：subaccount orphan 补内部储蓄
  - Step 7：未分类回滚 `is_pending=True` 重入收件箱
  - Step 8：cashflow 月份重算
- 跳过条件：`source='manual'` 或 `user_note IS NOT NULL`（用户手动行不被覆盖）
- 前端 `refresh-matching-button.tsx`（新文件）+ 浮窗汇总卡片
- 首次跑：清 21 个孤儿指针 + 自动配对 5 + 重判 116 个 type → transfer + 4 月重算

### 10. unbind-counter 端点 + 编辑表单显示对手账户
**问题**：编辑表单没有 counter_account_id 入口，"对方"字段是 counterparty 文本（PDF 几乎不填）。

**新增**：
- 后端 `POST /transactions/{id}/unbind-counter`：清自身指针；synthetic 对手→软删；real 对手→也清指针，对称解绑
- 前端 `transaction-form.tsx`：transfer 类型下显示对手账户名（只读）+ 「解除绑定」按钮
- 前端 `transaction-detail.tsx`：read-only view 加「对手账户」一行
- counterparty placeholder 改成「对方姓名/商户」消除歧义

### 11. delete_transaction 自动清对手指针
**问题**：软删一行不清对手 → 对手保留 stale counter_account_id → 「未配对」面板看不见。

**新增**：删除时检查 `metadata.paired_with_tx_id`，对手存在则清其 `counter_account_id` + paired_with_tx_id。
